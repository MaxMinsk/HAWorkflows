# Workflow.Web Frontend — Engineering Principles

Этот документ фиксирует принципы разработки frontend-части (`Vite + React + TypeScript`) и является основным reference для UI-реализации.

## 1) Цель frontend слоя

- Дать стабильный и предсказуемый UI для построения и запуска workflow.
- Сохранять код модульным: минимальный coupling, явные границы, простая поддержка.
- Поддерживать быструю итерацию (HMR/dev proxy) без деградации качества.

## 2) Технологический baseline

- Язык: только `TypeScript` (`.ts` / `.tsx`).
- UI: `React`.
- Build/dev: `Vite`.
- Type safety: `strict` mode обязателен.

## 3) Структура и границы

- Feature-first структура:
  - `app/` — композиция приложения.
  - `shared/` — переиспользуемые, доменно-нейтральные части.
  - `features/` — доменная логика по фичам (`workflow`, `runs`, `editor`).
- Направление зависимостей:
  - `shared -> features -> app`.
- Не связываем фичи напрямую друг с другом, если можно собрать их в `app`.

## 4) Архитектурные роли

- Components:
  - Только presentation/composition.
  - Никакой тяжелой бизнес-логики.
- Hooks:
  - Orchestration сценариев, state, effect lifecycle.
  - Не держат низкоуровневую инфраструктурную логику, если ее можно вынести.
- Adapters/Services (`features/*/lib`, `shared/lib`):
  - Инкапсулируют внешние детали (например, Drawflow API, localStorage).
  - Возвращают типизированные контракты без утечек внутренних деталей наружу.

## 5) TypeScript правила

- Публичные границы всегда типизированы:
  - props компонентов;
  - аргументы/returns hooks;
  - API payload/response;
  - adapter/service контракты.
- `any` не используем (допустим только в внешних `*.d.ts` для сторонних библиотек без типов).
- Для неизвестных данных используем `unknown` + явное сужение типа.
- Общие типы живут в `shared/types`.

## 6) Работа с состоянием и эффектами

- Локальное состояние остается локальным.
- Глобализуем состояние только при явной необходимости.
- Side effects держим в hooks, а не в компонентах.
- Эффекты должны иметь чистые зависимости и predictable cleanup.

## 7) Код-стиль и модульность

- Один модуль — одна ответственность.
- Один компонент/хук/adapter — один файл.
- Предпочитаем явные импорты; barrel используем точечно и осознанно.
- Повторяющуюся инфраструктурную логику выносим в shared helper/adapter.

## 8) Quality gates (обязательно)

Перед завершением задачи по frontend:

1. `npm run typecheck`
2. `npm run build`
3. (для интеграции проекта) `dotnet build WorkflowServer.sln`

## 9) Development flow

- Работаем по backlog и фиксируем выполненные задачи в `Done`.
- После изменений оставляем локальные сервисы запущенными для ручной проверки.
- Если static assets stale после пересборки Vite и `dotnet build` падает:
  - выполнить `dotnet clean src/Workflow.Web/Workflow.Web.csproj`;
  - затем повторить `dotnet build`.
