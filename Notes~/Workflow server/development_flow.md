# Development Flow (Workflow Server)

## 1) Цель

Зафиксировать единый процесс разработки, чтобы задачи выполнялись предсказуемо, а результат можно было быстро проверить локально.

## 2) Основные правила

1. Работаем по backlog (`P0 -> P1`), одну задачу за раз.
2. После завершения задачи обновляем `backlog.md`:
   - переносим задачу в `Done` (кратко),
   - обновляем `Now`.
3. Тесты не пишем, пока модуль/блок функционала не завершен полностью.
4. После завершения задачи обязательно запускаем локально и проверяем вручную.
5. После проверки оставляем сервис(ы) запущенными, чтобы пользователь мог сам проверить.
6. Если задача затрагивает UI, проверяем реальные пользовательские сценарии (не только сборку).
7. Любое отклонение от flow фиксируем в задаче кратким комментарием “почему”.

## 3) Порядок выполнения задачи

1. Взять задачу из `Now`.
2. Реализовать минимально необходимый scope (без лишнего refactor).
3. Проверить сборку/запуск локально.
4. Пройти ручной smoke-check по критериям задачи.
5. Оставить сервер(ы) поднятыми.
6. Обновить `backlog.md`.
7. Передать пользователю:
   - что сделано,
   - как открыть/проверить,
   - какой URL локального запуска.

## 4) Политика по тестам

- До завершения модуля: без тестов (ускоряем итерацию).
- После завершения модуля: добавляем тесты на ключевые сценарии и регрессии.
- При багфиксе в завершенном модуле: тест обязателен.

## 5) Локальная проверка (минимум)

- `dotnet build` проходит без ошибок.
- Нужные сервисы запускаются локально.
- Основной сценарий задачи воспроизводится вручную.
- Нет явных runtime ошибок в логах на стартовом пути.

## 6) Оставляем сервисы запущенными

После выполнения задачи сервисы не выключаем сразу, если пользователь не попросил остановить.
В ответе обязательно указываем:
- какой сервис запущен,
- на каком URL/порту,
- что именно проверить.

## 7) Артефакты после каждой задачи

Минимальный набор:
- изменение кода,
- обновленный `backlog.md`,
- краткий отчет “сделано/проверено/как проверить”.
